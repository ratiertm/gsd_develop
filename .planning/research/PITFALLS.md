# Pitfalls Research

**Domain:** 키움증권 OpenAPI+ 주식 데이트레이딩 자동매매 시스템
**Researched:** 2026-03-13
**Confidence:** MEDIUM (web search unavailable, based on training data for well-documented Korean fintech API domain)

## Critical Pitfalls

### Pitfall 1: TR 요청 제한(3.6초 룰) 위반으로 인한 연결 차단

**What goes wrong:**
키움 OpenAPI+는 TR(Transaction) 요청에 1초당 5회, 연속 조회시 약 3.6초 간격 제한이 있다. 이를 위반하면 서버에서 강제로 연결을 끊거나, 일정 시간 동안 요청이 차단된다. 장중 자동매매 중 연결이 끊기면 보유 포지션 관리가 불가능해져 큰 손실로 이어질 수 있다.

**Why it happens:**
- 여러 종목의 시세를 한꺼번에 조회하려고 for문으로 TR을 연속 호출
- 실시간 데이터와 TR 조회를 구분하지 않고, 실시간으로 받을 수 있는 데이터를 TR로 반복 조회
- 타이머 간격을 정확히 3.6초로 설정했지만, 시스템 지연으로 실제 간격이 더 짧아짐

**How to avoid:**
- TR 요청 큐를 만들어 중앙에서 요청 간격을 관리하는 RequestThrottle 클래스 구현
- 최소 간격을 4초로 여유있게 설정 (3.6초 + 안전 마진)
- 시세 데이터는 가능한 SetRealReg 실시간 등록으로 받고, TR은 과거 데이터/계좌 정보 등 꼭 필요한 경우에만 사용
- 연속 조회(sPrevNext)가 필요한 경우도 동일하게 간격 제한 적용

**Warning signs:**
- 에러코드 -200 (시세과부하) 발생
- 갑작스러운 연결 끊김 (OnDisconnect 이벤트)
- 장중 데이터 수신이 갑자기 멈춤

**Phase to address:**
Phase 1 (API 연동 기초) -- TR 요청 큐와 속도 제한 매니저를 아키텍처 초기부터 설계해야 함

### Pitfall 2: 주문 체결 확인 없이 다음 로직 진행 (비동기 주문 처리 실패)

**What goes wrong:**
SendOrder()는 주문 접수 결과만 즉시 반환하고, 실제 체결은 OnReceiveChejanData 이벤트로 비동기 수신된다. 주문 접수와 체결을 혼동하면:
- 같은 종목에 중복 주문 발생
- 미체결 주문이 남아있는데 추가 매수 실행
- 매도 주문이 체결되지 않았는데 포지션이 청산된 것으로 간주

**Why it happens:**
- SendOrder() 반환값 0(성공)을 "체결 완료"로 잘못 해석
- 이벤트 기반 프로그래밍에 익숙하지 않아 동기식으로 코드 작성
- 주문 상태 관리 상태머신(state machine) 없이 단순 플래그로 처리

**How to avoid:**
- 주문 상태를 명확히 관리: 접수요청 -> 접수확인 -> 체결(부분/전량) -> 완료
- OrderManager 클래스에서 주문번호(sOrderNo)를 키로 상태 추적
- OnReceiveChejanData에서 gubun(접수/체결/잔고)을 구분하여 처리
- 미체결 주문 목록을 주기적으로 조회(opt10075)하여 상태 동기화

**Warning signs:**
- 같은 종목에 동일 방향 주문이 여러 건 접수됨
- 계좌 잔고와 시스템이 추적하는 보유수량이 불일치
- 미체결 주문이 계속 누적됨

**Phase to address:**
Phase 2 (주문 실행 엔진) -- 주문 상태 관리가 자동매매의 핵심이므로, 매매 전략 구현 전에 반드시 완성

### Pitfall 3: PyQt5 이벤트 루프 블로킹으로 인한 시스템 프리즈

**What goes wrong:**
키움 OpenAPI+는 COM/OCX 기반으로 PyQt5의 이벤트 루프 위에서 동작한다. 무거운 계산(기술적 지표 계산, 백테스트)을 메인 스레드에서 수행하면 이벤트 루프가 블로킹되어:
- 실시간 데이터 수신 중단
- 키움 서버와의 heartbeat 실패로 연결 끊김
- GUI가 응답 없음(Not Responding) 상태로 전환
- 체결 이벤트를 놓쳐서 주문 상태 불일치

**Why it happens:**
- 키움 API 콜백이 반드시 메인 스레드에서 실행되어야 한다는 제약을 인지하지 못함
- time.sleep()이나 while 루프로 동기 대기 패턴 사용
- 멀티스레딩 시 키움 API를 서브 스레드에서 호출하려고 시도

**How to avoid:**
- 키움 API 호출과 이벤트 수신은 반드시 메인(QApplication) 스레드에서만 처리
- 무거운 계산은 QThread 또는 별도 프로세스로 분리하고, 결과를 시그널/슬롯으로 메인 스레드에 전달
- QEventLoop를 활용한 비동기 대기 패턴 사용 (time.sleep 금지)
- 지표 계산에 numpy/pandas 벡터 연산을 사용하여 계산 시간 자체를 최소화

**Warning signs:**
- GUI가 간헐적으로 멈추거나 응답 없음 표시
- 실시간 데이터 수신이 불규칙해짐
- 체결 이벤트가 지연되거나 누락됨

**Phase to address:**
Phase 1 (아키텍처 설계) -- 메인 스레드/워커 스레드 분리 구조를 초기부터 확립

### Pitfall 4: 리스크 관리 부재 또는 지연으로 인한 대규모 손실

**What goes wrong:**
손절매(stop-loss) 로직이 없거나, 있어도 제대로 작동하지 않는 경우:
- 급락장에서 포지션 전체가 하한가까지 하락
- 단일 종목에 자금이 집중되어 계좌 전체 큰 손실
- 일일 최대 손실 한도 없이 계속 매매하여 연쇄 손실

**Why it happens:**
- 전략 개발에 집중하느라 리스크 관리를 "나중에" 추가하려고 미룸
- 손절 로직은 있지만, 시장가 매도 주문이 미체결(호가 없음)인 경우를 처리하지 않음
- 트레일링 스탑의 기준가 업데이트가 실시간 데이터 수신과 동기화되지 않음
- 동시호가(08:30~09:00, 15:20~15:30) 시간에는 주문이 즉시 체결되지 않는 점을 고려하지 않음

**How to avoid:**
- 리스크 관리를 전략보다 먼저 구현: 최대 손실 한도, 종목별 비중 제한, 일일 손실 한도
- 계좌 잔고 대비 총 투입 비중 상한 설정 (예: 최대 60%)
- 단일 종목 최대 비중 제한 (예: 계좌의 10%)
- 손절 주문 미체결 시 재주문 로직 (가격 슬리피지 허용 확대)
- 하한가 매도 불가 상황 대비: 익일 장전 시간외 매도 예약

**Warning signs:**
- 백테스트에서 최대 낙폭(MDD)이 계좌의 30% 이상
- 단일 종목 보유 비중이 계좌의 20% 이상
- 손절 주문 후 실제 체결 확인 로직이 없음

**Phase to address:**
Phase 2 (주문 실행 엔진)와 Phase 3 (전략 엔진) 사이에 리스크 관리 레이어를 필수 구현

### Pitfall 5: 실계좌 테스트 없이 바로 실전 투입

**What goes wrong:**
모의투자 서버와 실서버의 동작 차이를 인지하지 못하고 바로 실계좌에서 운영:
- 모의투자에서는 모든 주문이 즉시 체결되지만, 실전에서는 호가 잔량에 따라 미체결/부분체결 발생
- 모의투자 서버의 TR 응답 형식이 실서버와 미세하게 다른 경우 존재
- 모의투자에서는 슬리피지가 없지만, 실전에서는 호가 스프레드로 예상과 다른 가격에 체결

**Why it happens:**
- 모의투자에서 잘 동작하는 것에 과신
- 실서버 전환 시 서버 주소(포트) 변경, 계좌번호 변경 등 설정 실수
- 모의투자 계좌와 실계좌의 API 동작 차이를 키움 공식 문서에서 명확히 안내하지 않음

**How to avoid:**
- 모의투자 -> 소액 실계좌(최소 수량) -> 본격 운영의 3단계 전환
- 실서버/모의서버 설정을 config 파일로 분리하고 환경 변수로 관리
- 실계좌 전환 초기에는 1주 단위 최소 수량으로 일주일간 운영
- 주문 실행 전 최종 확인 로그와 알림 (종목코드, 수량, 가격 등)

**Warning signs:**
- 모의투자에서만 백테스트/실전 테스트 완료
- 서버 전환 시 설정 변경 절차가 문서화되어 있지 않음
- 실계좌 전환 후 첫 주문에서 예상치 못한 체결가 발생

**Phase to address:**
최종 Phase (배포/운영) -- 모의투자 -> 소액 실전 -> 본격 운영 단계를 명시적으로 로드맵에 포함

### Pitfall 6: 키움 OpenAPI+ 로그인/세션 관리 실패

**What goes wrong:**
- 장중에 키움 서버 점검이나 네트워크 불안정으로 연결이 끊기면, 자동 재연결 로직이 없어서 매매가 중단됨
- 키움 OpenAPI+는 동시에 하나의 프로세스에서만 로그인 가능한데, 다른 프로세스(영웅문 HTS 등)가 로그인하면 기존 연결이 끊김
- 자동 로그인 설정을 하지 않아 매일 아침 수동으로 로그인 필요

**Why it happens:**
- 키움 API의 단일 세션 제한(1 IP 1 접속)을 인지하지 못함
- OnDisconnect/OnEventConnect 이벤트에 대한 핸들링 누락
- 키움 서버 정기 점검 시간(보통 장 종료 후)과 간헐적 장애에 대한 대비 부족

**How to avoid:**
- OnEventConnect 이벤트에서 연결 상태 모니터링 및 자동 재연결 로직 구현
- 재연결 시 기존 실시간 등록(SetRealReg) 목록을 복원하는 로직
- 키움 OpenAPI+ 로그인 시 자동 로그인 버전 관리 설정 활용
- 영웅문 HTS와 동시 사용 불가 안내 문서화
- 연결 상태를 주기적으로 GetConnectState()로 확인하는 heartbeat 구현

**Warning signs:**
- 간헐적으로 데이터 수신이 멈추지만 프로그램이 정상으로 보임
- GetConnectState()가 0(미연결) 반환
- 재시작 후 실시간 등록 종목이 초기화됨

**Phase to address:**
Phase 1 (API 연동 기초) -- 연결 관리를 가장 먼저 안정화해야 그 위에 모든 기능을 쌓을 수 있음

### Pitfall 7: 동시호가/장 시작-종료 시간대 미처리

**What goes wrong:**
- 09:00 이전 동시호가(08:30~09:00)에 주문을 넣었지만 09:00 시초가에 예상과 다른 가격에 체결
- 15:20 이후 장종료 동시호가에 매도를 시도했지만 주문 유형이 맞지 않아 거부
- 장 시작 직후(09:00~09:05) 변동성이 큰 시간에 대량 주문을 넣어 불리한 가격에 체결
- VI(변동성 완화장치) 발동 시 주문이 2분간 대기되는 것을 처리하지 않음

**Why it happens:**
- 한국 주식시장의 세부 시간 규칙을 코드에 반영하지 않음
- 동시호가와 접속매매의 주문 방식 차이를 구분하지 않음
- VI 발동, 서킷브레이커 등 특수 상황에 대한 처리 부재

**How to avoid:**
- 시장 시간 관리 모듈을 별도로 구현: 장전 동시호가, 정규장, 장후 동시호가, 시간외 단일가
- 매매 전략에 시간대별 제한 설정 (예: 09:00~09:05 매매 금지)
- VI 발동 감지 시 주문 대기 및 재시도 로직
- 장 종료 15분 전(15:15) 이후 신규 진입 금지, 기존 포지션 정리 모드 전환

**Warning signs:**
- 장 시작/종료 근처에서 예상치 못한 체결가
- 주문 거부 에러가 특정 시간대에 집중 발생
- VI 발동 시 주문 상태가 꼬임

**Phase to address:**
Phase 2 (주문 실행 엔진) -- 시장 시간 규칙을 주문 엔진에 내장

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| 전역 변수로 보유종목/잔고 관리 | 빠른 구현 | 동시성 문제, 상태 불일치, 디버깅 불가 | Never |
| TR 응답을 파싱하지 않고 문자열 슬라이싱 | 빠른 프로토타입 | 키움 API 응답 형식 변경 시 전면 수정 | MVP 초기 1주일만 |
| 에러 코드 무시하고 진행 | 코드가 단순해짐 | 장중 무음 실패(silent failure), 원인 추적 불가 | Never |
| 설정값 하드코딩 | 빠른 테스트 | 전략 변경 시 코드 수정 필요, 실수 유발 | Never -- 설정 파일은 처음부터 |
| 로그 없이 print()만 사용 | 빠른 디버깅 | 장중 문제 발생 시 사후 분석 불가 | 개발 초기 1일만 |
| 단일 파일에 모든 로직 | 초기 이해 용이 | 500줄 넘으면 유지보수 불가, 테스트 불가 | Never |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| 키움 OpenAPI+ COM/OCX | QAxWidget 생성을 서브스레드에서 시도 | 반드시 메인 스레드에서 생성, dynamicCall도 메인 스레드 |
| 키움 실시간 데이터 | SetRealReg에 종목 100개 이상 한번에 등록 | 종목 수 제한(약 100개), 필요 시 분할 등록 및 교체 |
| 키움 TR 데이터 | GetCommData 반환값에 공백 포함된 것 무시 | strip() 필수 적용, 숫자 변환 전 공백/부호 처리 |
| 키움 주문 | SendOrder의 sHogaGb(호가구분) 코드 오류 | 시장가="03", 지정가="00" 등 정확한 코드 사용, 매수/매도 구분(1/2) 확인 |
| 키움 에러코드 | 에러 반환값을 체크하지 않음 | SendOrder 반환값, OnReceiveTrData 에러 코드 모두 체크 |
| PyQt5 시그널/슬롯 | connect()를 여러 번 호출하여 슬롯이 중복 연결 | 연결 전 disconnect 시도하거나, 연결 관리 플래그 사용 |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| 실시간 데이터를 매 틱마다 DataFrame에 append | 장중 시간이 갈수록 메모리 사용량 증가, GUI 느려짐 | 고정 크기 deque/ring buffer 사용, 필요시만 DataFrame 변환 | 종목 10개 이상, 장 중반(11시) 이후 |
| 매 틱마다 기술적 지표 전체 재계산 | CPU 사용률 급증, 이벤트 처리 지연 | 증분(incremental) 지표 계산, 또는 N틱/N초마다 배치 계산 | 종목 5개 이상 동시 모니터링 |
| matplotlib로 실시간 차트 업데이트 | 차트 갱신이 이벤트 루프 블로킹 | pyqtgraph 사용 (GPU 가속, Qt 네이티브), 또는 차트 업데이트 주기 제한(1초) | 차트 1개라도 매 틱 갱신 시 |
| 모든 체결/호가 데이터를 DB에 실시간 INSERT | 디스크 I/O 병목, 주문 처리 지연 | 메모리 버퍼에 모아서 일괄 INSERT (1초/5초 단위), 또는 장 종료 후 일괄 저장 | 종목 20개 이상 |
| 과거 데이터 조회를 장중에 대량 실행 | TR 요청 제한에 걸려 실시간 매매에 필요한 TR 사용 불가 | 과거 데이터는 장 시작 전에 미리 조회/캐싱, 장중에는 실시간 데이터만 사용 | TR 요청이 5건/초 이상 |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| 계좌 비밀번호를 소스코드에 하드코딩 | Git 커밋 시 비밀번호 노출, 계좌 탈취 위험 | 환경변수 또는 암호화된 설정 파일 사용, .gitignore에 설정 파일 추가 |
| 키움 자동 로그인 비밀번호를 평문 저장 | 파일 접근 시 계좌 비밀번호 노출 | Windows 자격 증명 관리자 또는 keyring 라이브러리 활용 |
| 매매 로그에 계좌번호 전체 기록 | 로그 파일 유출 시 계좌 특정 가능 | 계좌번호 마스킹 (앞 4자리만 표시), 로그 파일 접근 권한 제한 |
| 주문 수량/금액에 대한 상한 검증 없음 | 코드 버그로 비정상적 대량 주문 발생 | 주문 전 최대 수량/금액 하드 리밋 검증, 비정상 주문 차단 |
| Discord 웹훅 URL을 소스코드에 포함 | 웹훅 URL 노출 시 스팸 메시지 가능 | 환경변수로 관리, .env 파일 사용 |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| 매매 상태를 콘솔 로그로만 표시 | 장중 실시간 모니터링 불가, 문제 인지 지연 | GUI 대시보드에 주문 상태, 보유종목, 손익 실시간 표시 |
| 긴급 매도(전량 청산) 버튼 없음 | 비상 시 수동 개입 불가, HTS를 별도로 열어야 함 | 긴급 전량 매도 버튼 + 전략 일시정지 버튼 GUI에 배치 |
| 전략 파라미터 변경 시 프로그램 재시작 필요 | 장중 전략 조정 불가 | 런타임 설정 변경 기능 + 변경 이력 로깅 |
| 에러 발생 시 사용자에게 알림 없음 | 연결 끊김/주문 실패를 모르고 방치 | 에러 등급별 알림: GUI 팝업(긴급) + Discord(중요) + 로그(일반) |
| 백테스트 결과를 숫자로만 표시 | 전략 성과 직관적 파악 불가 | 수익곡선, MDD 차트, 월별 수익률 히트맵 등 시각화 |

## "Looks Done But Isn't" Checklist

- [ ] **주문 실행:** 시장가 주문이 잘 되지만, 미체결/부분체결/주문취소/주문정정 처리가 없음 -- 전체 주문 상태 머신 구현 확인
- [ ] **실시간 데이터:** 데이터 수신은 되지만, 연결 끊김 후 재등록 로직이 없음 -- 연결 복구 후 SetRealReg 재호출 확인
- [ ] **손절매:** 손절 로직이 있지만, 하한가/호가 없음 상황에서 미체결 처리가 없음 -- 손절 미체결 시 재주문 로직 확인
- [ ] **백테스트:** 수익률은 나오지만, 슬리피지/수수료/세금 미반영 -- 왕복 수수료 0.015%+세금 0.23%(코스피)/0.23%(코스닥) 반영 확인
- [ ] **리스크 관리:** 종목별 손절은 있지만, 계좌 전체 일일 손실 한도가 없음 -- 일일 최대 손실 도달 시 전략 중지 확인
- [ ] **시간 관리:** 장중 매매는 되지만, 장 종료 전 미체결 주문 자동 취소가 없음 -- 15:20 전 미체결 정리 로직 확인
- [ ] **로깅:** 로그는 남기지만, 주문번호/체결번호와 연계되지 않아 추적 불가 -- 주문 추적 ID 기반 로그 확인
- [ ] **설정 관리:** 전략 파라미터가 있지만, 변경 이력이 없어 어떤 설정으로 어떤 결과가 나왔는지 모름 -- 설정 스냅샷 저장 확인

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| TR 요청 과부하로 연결 차단 | LOW | 프로그램 재시작, 1분 대기 후 재연결, 미체결 주문 확인 |
| 중복 주문 발생 | MEDIUM | 미체결 주문 전량 취소(opt10075 조회 후), 보유잔고 확인, 초과 매수분 시장가 매도 |
| 이벤트 루프 블로킹으로 데이터 손실 | MEDIUM | 프로그램 재시작, 실시간 등록 복원, 누락된 데이터 TR로 보충 조회 |
| 리스크 한도 초과 손실 | HIGH | 전량 청산 후 원인 분석, 리스크 파라미터 재설정, 소액으로 재개 |
| 모의투자/실계좌 설정 혼동 | HIGH | 즉시 전략 중지, 실계좌 주문 확인 및 필요 시 수동 정리, 설정 분리 개선 |
| 장 종료 시 미체결 주문 방치 | MEDIUM | 익일 장전 시간외 매매로 정리, 미체결 자동 취소 로직 추가 |
| 세션 끊김 중 포지션 보유 | HIGH | HTS(영웅문)로 수동 확인 및 매도, 자동 재연결 및 포지션 동기화 구현 |
